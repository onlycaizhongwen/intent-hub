1. 接入与治理层 (Traffic Gateway)

定位：系统的“城门”与“安检口”。

核心职责：

身份鉴权：验证 API Key、JWT，区分第三方系统与 C 端用户。

流量治理：限流（Rate Limit）、熔断（Circuit Breaker）、降级（Degradation）。

安全防护：WAF 防火墙，拦截 SQL 注入、Prompt 注入及恶意攻击。

日志监控：全链路 TraceID 生成，指标上报（Prometheus）。

2. 输入适配层 (Input Adapter)

定位：协议的“万能插头”。

核心职责：

多源归一：将 Web Chat、App、第三方 API、IoT 设备的异构输入，统一封装为标准 Envelope（信封）​ 格式。

会话加载：根据 session_id从 Redis 加载多轮对话上下文。

协议转换：将非文本信号（如按钮点击、系统事件）转换为文本意图信号。

3. 核心意图漏斗 (NLU Core Pipeline)

定位：系统的“大脑皮层”。

核心职责：

预处理（Preprocessing）：文本清洗、纠错、实体掩码（Masking）、同义词归一。

规则引擎（Rule Engine）：基于 AC 自动机/正则的高频意图快速匹配（毫秒级）。

BERT 精排（BERT Classifier）：基于 DeBERTa/ERNIE 的千级意图分类，输出高置信度结果。

LLM 兜底（OddAgent）：处理长尾、复杂语义及指代消解，作为最后一道防线。

拒识模块（Out-of-Scope）：拦截未知意图与越狱行为，防止幻觉。

4. 场景路由层 (Scene Router)

定位：系统的“导航仪”。

核心职责：

意图接收：接收来自核心漏斗的标准 JSON 输出（Intent + Slots + Confidence）。

场景匹配：依据 Scene Routing Table（场景路由表），根据 source、user_tags等元数据，匹配对应的业务场景（如电商、金融、IoT）。

策略分发：为不同场景加载特定的 NLU 策略（如模型版本、置信度阈值、是否启用 LLM）。

5. 输出适配层 (Output Adapter)

定位：下游业务的“翻译官”。

核心职责：

标准化输出：生成统一 JSON（Intent Code, Slots, Confidence, Debug Info）。

业务映射：根据场景路由表，将意图映射为下游具体的 API 调用、MQTT 指令或数据库操作。

幂等控制：生成 idempotency_key，确保业务执行的原子性。

6. 配置与数据闭环层 (Ops & Data)

定位：系统的“免疫系统”与“进化引擎”。

核心职责：

配置中心：热更新意图库、同义词表、场景路由规则（Nacos/Apollo）。

Bad Case 回流：收集低置信度、人工转接的日志，形成标注数据集。

模型迭代：基于回流数据周期性 Retrain BERT 模型，优化 Prompt。

数据流向总结

输入：[User/API] -> Gateway (Auth/Log) -> Adapter (Envelope)

决策：Router (Match Scene) -> Load Strategy (Model/Threshold)

计算：Preprocess -> Rule -> BERT -> LLM -> OutOfScope

输出：Standard JSON -> Executor (Downstream API) -> Response

进化：Feedback Log -> Annotation -> Retrain

设计理念一句话总结

“协议无关的输入，场景自适应的计算，标准确定的输出。”

这套分层架构确保了 IntentHub​ 在高并发、多业务、低延迟的生产环境下，具备极强的 鲁棒性（Robustness）​ 与 可扩展性（Scalability）。



要求适配场景：1.场景一：多轮对话（Chat/IVR），支持用户与助手的多轮交互，如订单查询、商品推荐等。
            2.场景二：单轮对话（如语音识别），支持用户直接输入文本，如查询天气、翻译等。
            3.场景三：第三方接口数据（API/Webhook），如支付、订单、库存等。
            4.场景四：自定义场景，根据业务需求定制意图识别规则。
            5.场景五：多模态输入（如图片、视频），支持用户通过非文本信号（如点击、系统事件）触发意图识别（最后做，预留选项）。
