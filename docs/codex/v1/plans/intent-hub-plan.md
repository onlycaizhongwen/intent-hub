# 意图中枢实施计划

## 目标

按“先闭环、再增强、后平台化”的顺序建设意图中枢。第一阶段先证明标准输入、识别漏斗、场景路由、标准输出、bad case 回流可以跑通。

## 前置条件

- 明确首批业务场景 2-3 个，例如订单查询、订单取消、库存事件。
- 提供首批意图与槽位清单。
- 确认部署基线：Java/Spring 版本、数据库、Redis、MQ、网关、日志平台。
- 明确模型来源：已有分类模型、待训练模型，或先用模拟模型接口。
- 明确 LLM provider、预算、安全边界和是否允许外部调用。
- 已确认风险修正事项作为 v1 强制约束，尤其要突出三大核心变更点：双阶段路由、LLM 受控、防腐层。
- 双阶段路由：先选怎么认，再选谁来干。
- LLM 受控：最后一道防线，不是主力。
- 防腐层：输出适配层只发指令，不碰业务数据。

## 阶段 0：项目骨架与契约

- 建立模块结构：API、core、config、model-client、router、adapter、admin、worker。
- 定义 Envelope、IntentResult、IntentDefinition、SceneRoutingRule 等核心 DTO。
- 定义错误码、拒识码、debug 字段、TraceID 规范。
- 设计 DB Schema：配置域表、运行域表、bad case、audit_log、idempotency_record。
- 定义 `decision` 枚举：`SUCCESS`、`CLARIFY`、`REJECTED`、`HANDOFF`、`BLOCKED`、`ASYNC_ACCEPTED`。
- 准备 OpenAPI 文档和示例请求。
- 固化 P0 评审微调：attachments 只存 URL/Object Key、confidence 范围、input_snapshot 脱敏、match_condition 示例、LLM `max_retries`、downstream `timeout_ms`。

验证方式：

- 契约样例通过 schema 校验。
- 单元测试覆盖 DTO 序列化、字段兼容性和错误码。
- DB Schema 评审通过，确认 `数据流向图v2.png` 作为处理与回流基线，表结构仍以 P0 DB Schema 为准。
- P0 评审结论为 Approved，可以进入 P1。

## 阶段 1：最小识别闭环

- 实现 REST 识别入口。
- 实现输入适配和 Redis 会话读取。
- 实现预处理：清洗、同义词归一、基础脱敏。
- 实现规则引擎最小版：关键词、正则、优先级、命中解释。
- 实现场景路由表读取和匹配。
- 实现双阶段路由：前置选策略，后置选业务动作。
- 实现槽位状态机最小版：未填、已抽取、待确认、已确认、已过期。
- 实现标准输出。
- 建立配置治理最小方案：Admin Portal 或 GitOps 二选一，禁止手工改数据库作为正式配置发布方式。

下一步执行详见 `docs/codex/v1/plans/intent-hub-p1-next-step-plan.md`。P1 当前不应直接跳到 P2，优先把内存版闭环推进为可编译、可验收、可回溯、可配置发布的工程闭环。

验证方式：

- 3 个业务意图可通过规则识别返回结果。
- 缺失槽位可返回澄清动作。
- 低置信度或未命中可拒识。
- `decision` 字段可表达澄清、拒识、人工转接和阻断。
- 配置变更可审计、可回滚，并可同步到目标环境。

### P1 高风险点：配置管理一致性

P1 不允许依赖手工改数据库同步配置。必须选择至少一种配置治理方案：

- Admin Portal：通过管理后台维护、发布、回滚配置。
- GitOps：配置文件版本化，经过评审后发布到环境。

该风险来自 P0 评审，优先级高于普通功能增强。

## 阶段 2：模型分类与 LLM 兜底

- 抽象模型分类接口。
- 接入模型服务或模拟模型服务。
- 定义候选意图融合策略：规则、模型、LLM 的优先级和置信度合并。
- 接入 LLM 兜底接口，设置开关、超时、预算和审计。
- 加入 Prompt 注入检测和敏感字段脱敏。
- 增加 LLM 熔断、降级和关闭策略。

验证方式：

- 模型候选与规则候选冲突时按策略可解释地选择。
- LLM 超时或失败时可降级。
- LLM 调用日志可追踪到策略、输入摘要和输出结果。
- LLM 不在主识别路径默认触发。

## 阶段 3：配置管理与发布

- 建立意图、槽位、同义词、场景、策略、路由表。
- 实现 `tenant_id + scene_id + version` 配置版本、发布、回滚。
- 实现本地缓存和配置变更通知。
- 接入 Nacos/Apollo 可作为后续增强，v1 可先以数据库版本发布实现。

验证方式：

- 修改同义词或规则后发布新版本，识别结果按版本生效。
- 回滚配置后识别结果恢复。
- 配置发布有审计记录。
- 租户 A 的配置变更不影响租户 B。

## 阶段 4：反馈闭环与观测

- 记录 RecognitionTrace。
- 记录 bad case：低置信度、拒识、人工修正、下游失败。
- 输出 Prometheus 指标。
- 接入 OpenTelemetry trace。
- 准备 bad case 导出格式，供标注和训练使用。
- 记录 `idempotency_record`，保存下游动作状态、请求摘要、重试次数和过期时间。

验证方式：

- 任一请求可按 trace_id 查到完整链路。
- bad case 可按租户、场景、意图、错误类型筛选。
- 指标可展示规则命中率、模型命中率、LLM 调用率、拒识率。
- 重复请求不会重复触发下游扣款、下单等动作。

## 阶段 5：生产化增强

- 增加 API Key/JWT 鉴权。
- 增加租户隔离和 RBAC。
- 增加限流、熔断、降级。
- 增加异步执行通道和幂等状态表。
- 增加灰度模型、灰度配置和 A/B 对比。
- 明确同步识别与异步执行业务边界，异步通过 MQ/Callback/Webhook。

验证方式：

- 鉴权失败、限流、降级都有明确响应。
- 同一 idempotency_key 不会重复触发下游动作。
- 灰度租户只命中新模型或新配置。
- Output Adapter 不存在通用直写业务数据库能力。

## 检查点

- P0：契约评审通过。
- P1：规则识别闭环演示通过。
- P2：模型和 LLM 融合策略评审通过。
- P3：配置发布与回滚演示通过。
- P4：可观测和 bad case 回流演示通过。
- P5：生产安全评审通过。

## 回滚策略

- 配置回滚：所有配置发布保留版本，支持按租户和场景回滚。
- 模型回滚：模型版本通过策略配置引用，不在代码中硬编码。
- LLM 降级：LLM provider 异常时关闭兜底，仅保留规则和模型路径。
- 下游执行回滚：v1 优先通过业务 API/MQ，避免直接写业务库；失败时按业务幂等键重试或人工处理。

## 建议里程碑

- 第 1 周：契约、表结构、规则识别最小闭环。
- 第 2 周：场景路由、输出适配、会话上下文、拒识。
- 第 3 周：模型接口、LLM 兜底、融合策略。
- 第 4 周：配置发布、bad case、观测、安全加固。
- 第 5 周：压测、灰度、试点场景上线。
