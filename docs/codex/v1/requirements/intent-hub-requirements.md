# 意图中枢需求文档

## 背景

意图中枢（IntentHub）用于把来自 Chat、IVR、App、第三方 API/Webhook、IoT 事件等多源输入，统一识别为标准意图结果，并按业务场景路由给下游系统执行。现有资料已经给出六层架构：接入与治理、输入适配、核心意图漏斗、场景路由、输出适配、配置与数据闭环。

本阶段目标不是直接实现完整平台，而是明确 v1 的可交付范围、边界、技术路线和落地计划。

## 目标

- 支持协议无关输入：统一封装文本、事件、Webhook 等输入为标准 Envelope。
- 支持多场景意图识别：多轮对话、单轮文本、第三方接口数据、自定义场景优先；多模态作为预留能力。
- 支持混合 NLU 策略：规则快速命中、模型分类、LLM 兜底、拒识保护。
- 支持场景化路由：按 source、tenant、user_tags、scene、intent、confidence 等元数据匹配业务场景和策略。
- 支持标准输出：统一输出 intent、slots、confidence、scene、trace、debug、idempotency_key。
- 支持配置热更新和数据闭环：意图库、同义词、路由规则、阈值、模型版本可配置；低置信度和 bad case 可回流。

## 三大核心变更点

这三点是意图中枢区别于普通 NLU 系统的核心需求：

- 双阶段路由：先选怎么认，再选谁来干。前置路由选择识别策略，后置路由选择业务动作。
- LLM 受控：LLM 是最后一道防线，不是主力。仅在规则和模型无法稳定处理时触发，必须具备开关、超时、预算、熔断、降级、注入防护和审计。
- 防腐层：输出适配层只发指令，不碰业务数据。v1 仅允许 API、MQ、Webhook、MQTT 等契约交互，严禁通用直写业务数据库。

## 用户与系统角色

- C 端用户：通过 Chat、IVR、App 等入口发起自然语言或事件交互。
- 第三方系统：通过 API/Webhook 推送结构化或半结构化事件。
- 业务系统：接收标准意图结果或执行指令，例如订单、支付、库存、IoT 控制。
- 运营/标注人员：维护意图库、规则、同义词、路由策略，处理 bad case。
- 平台管理员：管理租户、鉴权、限流、模型版本、观测和告警。

## 范围

### v1 必做

- 标准 Envelope 输入模型。
- REST API 接入，预留 Webhook 接入。
- Redis 会话上下文加载与保存。
- 规则引擎：关键词、正则、AC 自动机或 Trie 类高频匹配。
- 模型分类接口抽象：先接入本地/远程分类服务，具体模型可替换。
- LLM 兜底接口抽象：作为最后一道防线而非主力，支持开关、超时、预算、熔断、降级、审计和 Prompt 注入防护。
- 拒识机制：低置信度、越界意图、敏感/危险请求拦截，并通过标准 `decision` 输出 `REJECTED`、`HANDOFF`、`CLARIFY`、`BLOCKED` 等结果。
- 双阶段场景路由：前置路由按 `tenant_id/source/channel` 选择识别策略；后置路由按 `intent/slots/confidence` 选择业务动作。
- 标准输出 JSON。
- 基础配置中心能力：先以数据库 + 本地缓存实现，后续可接 Nacos/Apollo；全链路配置按 `tenant_id + scene_id + version` 管理并支持回滚。
- 基础可观测：TraceID、结构化日志、核心指标。
- Bad Case 记录：保存原始输入、识别链路、候选结果、最终结果、错误原因。
- DB Schema：补齐配置域、运行域、bad case、审计和幂等记录表。
- 槽位生命周期：支持未填、已抽取、待确认、已确认、已过期状态，支持缺槽追问、上下文继承、校验和归一化。
- 同步识别与异步执行：同步链路返回识别结果；异步链路通过 MQ/Callback/Webhook 执行业务动作。
- 幂等与重试：下游动作必须携带幂等键并记录执行状态，防止重复扣款、重复下单等重复执行风险。

### v1 可选

- MQ 异步执行通道。
- 管理后台最小页面。
- 向量库用于同义表达、示例检索或少样本匹配。
- 灰度策略：按租户、场景、模型版本灰度。

### 暂不做

- 图片、视频等完整多模态识别。
- 自动训练平台和完整 MLOps 流水线。
- 复杂工作流编排引擎。
- 直接写业务数据库的通用执行器；输出适配层作为防腐层，只发指令，不碰业务数据。v1 严禁通用直写业务库，仅允许通过 API、MQ、Webhook、MQTT 等契约交互。
- 高阶对话管理，例如复杂任务规划、多 Agent 协作。

## 关键场景

### 多轮对话

用户通过 Chat/IVR 发起请求，系统按 session_id 加载上下文，结合当前输入识别意图和槽位。若槽位缺失，输出澄清动作；若满足执行条件，路由到业务系统。

### 单轮文本

用户直接输入一次性请求，例如天气查询、翻译、订单查询。系统不依赖历史上下文即可完成识别和路由。

### 第三方 API/Webhook

外部系统推送支付、订单、库存等事件。输入适配层把结构化事件转为标准 Envelope，意图中枢识别事件语义并路由给业务处理方。

### 自定义场景

运营或业务方维护场景规则、意图库、阈值和下游动作映射。系统支持热更新和版本化审计。

### 多模态预留

v1 仅保留 `input_type`、`attachments`、`metadata` 等预留字段，不实现图片/视频语义识别或解析。

## 输入输出要求

### 输入 Envelope

最小字段：

- `request_id`
- `tenant_id`
- `source`
- `channel`
- `session_id`
- `user_id`
- `input_type`
- `text`
- `event`
- `metadata`
- `timestamp`

### 输出 IntentResult

最小字段：

- `request_id`
- `trace_id`
- `tenant_id`
- `scene_id`
- `intent_code`
- `intent_name`
- `slots`
- `confidence`
- `recognition_path`
- `decision`
- `idempotency_key`
- `debug`
- `errors`

## 非功能需求

- 延迟：规则命中链路 P95 优先控制在 50ms 内；模型分类链路 P95 优先控制在 300ms 内；LLM 兜底链路独立设置超时。
- 可用性：核心识别服务和配置读取必须支持降级；LLM 不可用时不能阻断规则/模型链路。
- 安全：鉴权、限流、租户隔离、Prompt 注入防护、敏感信息脱敏、审计日志必须纳入 v1。
- 可观测：请求必须带 TraceID；记录分层耗时、命中策略、模型版本、置信度、拒识原因。
- 可扩展：新增渠道、新增场景、新增模型、新增下游动作不应修改核心识别流程。
- 可回溯：每次意图结果应可追踪到输入、配置版本、模型版本和路由版本。
- 幂等性：触发下游业务动作时必须通过 `idempotency_key` 和执行记录防止重复执行。

## 完成标准

- 形成可评审的需求、技术方案、实施计划和资料审查文档。
- v1 范围和暂不做范围清晰。
- 关键输入输出协议有最小字段定义。
- 技术选型有主选和备选，并说明取舍。
- 原始资料中不合理或不清晰之处已列出。
- 用户确认的风险修正事项已固化到需求、设计、计划、审查和 HTML 阅读页。

## 依赖

- 业务方提供首批意图库、槽位定义、场景路由规则和下游动作契约。
- 运维方明确部署环境、网关选型、注册配置中心、日志指标平台。
- 算法/模型方提供分类服务接口、训练数据格式、模型版本策略。
