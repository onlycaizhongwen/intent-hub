## v1 强制约束

- 三大核心变更点必须作为评审主线：双阶段路由、LLM 受控、防腐层。
- `数据流向图v2.png` 是当前数据流向基线；`数据流向图v1.png` 仅作为历史材料保留。
- 数据库设计必须覆盖配置版本、识别日志、bad case 和幂等记录。
- 场景路由必须实现前置路由和后置路由两个概念，不允许只用单点路由表达所有逻辑。
- LLM 调用必须受策略控制，必须具备超时、预算、熔断、降级、注入防护和审计。
- Output Adapter 是防腐层，只发指令，不碰业务数据，不得作为通用业务数据库写入器。
- 所有配置必须带租户、场景、版本和发布审计信息。
- 槽位必须具备生命周期状态，不能只作为普通 key-value。
- 拒识、澄清、人工转接、阻断必须通过标准 `decision` 表达。
- 同步识别与异步业务执行必须在接口和时序上分开。
- 下游动作必须具备幂等键和重试状态。
- 多模态字段仅作为兼容预留，不进入首期验收。

## DB Schema 规划

### 配置域

- `tenant`：租户基础信息。
- `scene`：业务场景定义。
- `config_version`：配置版本、发布状态、发布人、发布时间、回滚来源。
- `intent_definition`：意图定义、样例、状态、版本。
- `slot_definition`：槽位定义、类型、校验、归一化策略。
- `synonym_mapping`：同义词和归一化词表。
- `nlu_strategy`：识别策略，包含规则集、模型版本、LLM 开关、阈值。
- `scene_routing_rule`：前置/后置路由规则、优先级、下游动作。

### 运行域

- `conversation_session`：会话上下文索引和过期时间。
- `slot_state`：槽位生命周期，包含未填、已抽取、待确认、已确认、已过期。
- `recognition_trace`：识别链路、候选结果、最终决策、版本信息、分层耗时。
- `bad_case`：低置信度、拒识、人工修正、下游失败样本。
- `idempotency_record`：幂等键、请求摘要、下游动作、执行状态、重试次数、过期时间。
- `audit_log`：配置发布、模型切换、阈值调整、人工干预记录。
