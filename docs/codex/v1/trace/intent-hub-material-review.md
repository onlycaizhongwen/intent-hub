# 意图中枢资料审查与不合理点

## 审查来源

- `docs/codex/v1/designs/意图中枢需求说明.md`
- `docs/codex/v1/designs/意图中枢架构图.png`
- `docs/codex/v1/designs/数据流向图v1.png`
- `docs/codex/v1/designs/数据流向图v2.png`
- `docs/codex/v1/designs/核心交互时序图.png`

## 总体判断

现有资料方向正确，适合作为意图中枢的概念架构：分层清晰，覆盖接入治理、输入适配、意图识别、场景路由、输出适配、数据闭环。`数据流向图v2.png` 已修正 v1 中对路由位置和数据回流表达不够清晰的问题，可作为当前数据流向基线。但资料仍偏概念化，缺少 v1 边界、数据模型、接口契约、配置发布、失败补偿和安全细节。用户已确认风险修正方案，后续以 `intent-hub-confirmed-decisions.md` 中的确认项作为 v1 规划约束，并重点突出三大核心变更点：双阶段路由、LLM 受控、防腐层。

## 需要修正或澄清的问题

### 1. v1 图命名和表达不准确，v2 已修正

v1 图展示的是请求处理流和数据回流，不是数据库表、数据存储或读写关系。当前已采纳 `数据流向图v2.png` 作为数据流向基线，真实数据库设计仍以 P0 DB Schema 为准，覆盖意图表、槽位表、场景路由表、策略表、识别日志、bad case 表、配置版本表、幂等记录表等。

### 2. 场景路由位置表达不一致，v2 已拆成双阶段路由

文字中“数据流向总结”写为 Router 先匹配场景再加载策略，然后执行 Preprocess/Rule/BERT/LLM；架构图则把场景路由层画在核心意图漏斗上方，但时序图中核心意图漏斗先识别，再把结果交给场景路由。

已确认拆成两段，且 v2 图已显式体现：

- 前置路由：按 tenant/source/channel 初筛策略。
- 后置路由：按 intent/slots/confidence 决定下游动作。

### 3. LLM 兜底边界过宽

资料把 LLM 作为最后一道防线，但没有说明触发条件、超时、成本预算、审计、Prompt 注入防护和降级策略。已确认 LLM 仅作为受控兜底，必须配置熔断、降级、注入防护、超时、预算和审计。

### 4. 输出适配层“数据库操作”风险较高

资料提出输出适配可映射为 API 调用、MQTT 指令或数据库操作。已确认 v1 严禁通用直写业务数据库，仅允许通过 API、MQ、Webhook、MQTT 等契约交互。

### 5. 缺少租户与配置版本概念

意图库、同义词、路由规则、模型版本、阈值已确认按 `tenant_id + scene_id + version` 管理，支持发布审计和回滚。

### 6. 缺少槽位生命周期

现有资料提到 Slots，但没有定义槽位抽取、校验、归一、缺失追问、上下文继承和过期策略。已确认槽位生命周期纳入 v1，包含未填、已抽取、待确认、已确认、已过期等状态。

### 7. 缺少拒识和人工兜底策略

资料提到 Out-of-Scope，但没有明确拒识返回结构、人工转接、澄清提问、低置信度阈值、越界策略和越狱攻击处置。已确认通过标准 `decision` 字段输出 `REJECTED`、`HANDOFF`、`CLARIFY`、`BLOCKED` 等结果。

### 8. 时序图响应链路过于简化

时序图第 8 步从业务执行层直接返回到客户端，跨过了场景路由、核心意图漏斗、输入适配和网关。已确认区分同步识别与异步执行，异步通过 MQ/Callback/Webhook。

### 9. 缺少幂等和重试存储

资料提到 `idempotency_key`，但没有说明如何生成、保存、判重、过期和与下游重试关联。已确认补 `idempotency_record`，防止下游重复执行。

### 10. 多模态不应进入 v1 主线

多模态输入复杂度高，涉及文件存储、内容安全、OCR/ASR/CV 模型和异步处理。已确认 v1 延期，仅预留字段，不做实际解析。

## 建议优先补充的资料

- 标准 Envelope 协议。
- IntentResult 协议。
- 意图、槽位、场景、策略、路由配置模型。
- 识别链路决策规则：规则、模型、LLM、拒识的优先级。
- 配置发布和回滚流程。
- Bad Case 标注字段。
- 可观测指标和审计字段。
- 下游动作契约和幂等策略。
- 用户确认决策清单：`docs/codex/v1/trace/intent-hub-confirmed-decisions.md`。

## 结论

现有资料可以作为概念方案，但不能直接作为开发需求。建议以本次新增的 requirements、design、plan 作为 v1 落地基线，再逐步把原始图中的长期能力拆到后续版本。
